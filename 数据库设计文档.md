# 医疗预约挂号系统（Web端）数据库设计文档（MySQL）

- 项目名称：医疗预约挂号系统（PC Web）
- 技术栈：Vue3 + Spring Boot + MySQL
- 文档版本：V1.0
- 数据库：MySQL 8.x（建议）
- 字符集/排序规则：utf8mb4 / utf8mb4_general_ci
- 存储引擎：InnoDB

> 本文档用于描述系统数据库的概念设计、逻辑设计、物理表结构、约束与索引策略，以及关键业务（预约扣减、冲突检测、签到排队、调班替诊、评分与统计）在数据库层面的实现落点，为后续编码与测试提供依据。

---

## 1. 设计原则

### 1.1 规范性与一致性
1. 表名采用小写下划线风格（snake_case），如 `sys_user`、`doctor_profile`。
2. 主键统一使用 `BIGINT` 自增（便于毕设演示与排错）。
3. 时间字段统一：`created_at`、`updated_at`，必要时使用 `*_at` 表示业务时间点（如 `paid_at`、`checked_in_at`）。
4. 枚举字段使用 `VARCHAR(64)` 存储，**关联字典项编码**（便于阅读、扩展与前端通用化配置），在数据库层面通过外键约束保证合法性。

### 1.2 性能与可扩展
1. 针对高频查询（医生筛选、号源查询、预约冲突检测、排队队列查询）建立组合索引。
2. 预约相关关键表采用冗余字段（如 `appointment` 冗余 `visit_date`、`time_slot`、`dept_id`），减少多表 join，提高统计与筛选性能。
3. 统计分析提供两种实现策略：实时聚合（默认）与每日汇总表（可选扩展）。
4. **通过字典表支持动态基础数据**：系统状态（预约状态、排队状态）、职称等通过字典管理，管理员可灵活修改显示名称或新增条目。

## 2. 概念结构设计（ER 关系说明）

### 2.1 核心实体
- 科室（Department）
- 用户账户（User）
- 患者档案（PatientProfile）
- 医生档案（DoctorProfile）
- 排班与号源（ScheduleSlot）
- 预约订单（Appointment）
- 支付记录（PaymentRecord）
- 退款记录（RefundRecord）
- 签到与排队（CheckinQueue / QueueState）
- 调班申请（ShiftRequest）
- 评分记录（DoctorRating）
- 提醒日志（NotifyLog）
- 统计汇总（StatDailyDoctor，可选）
- **数据字典类型（DictType）**
- **数据字典项（DictItem）**

### 2.2 实体关系（文字版）
1. 科室 1—N 医生：一个科室可包含多个医生。
2. 用户 1—1 患者档案/医生档案：用户账号与具体业务身份关联。
3. 医生 1—N 排班号源：每名医生可配置未来多天上午/下午排班与容量。
4. 患者 N—N 医生：通过预约订单建立联系（一次预约只对应一个医生与一个时段）。
5. 预约 1—0/1 支付与退款：一条预约可能产生支付和退款记录。
6. 预约 1—0/1 签到排队：只有签到后才产生排队记录。
7. 预约 1—0/1 评分：就诊完成后每单仅允许评分一次。
8. 调班申请关联排班，并指定接替医生（或由管理员后续指派）。
9. **字典类型 1—N 字典项**：一种类型（如“预约状态”）对应多个具体的项（如“待支付”、“已完成”）。
10. **业务表 N—1 字典项**：业务表中的状态/类型字段通过 `item_code` 关联到唯一的字典项。

## 3. 逻辑结构设计（业务规则落点）

### 3.1 号源与排班模型
- 系统按“日期 + 上午/下午 + 医生”管理号源：
  - `schedule_slot`：承载容量 `capacity` 与剩余量 `remaining`，以及费用 `fee`、状态 `status`。
- 唯一约束：
  - 同一医生同一日期同一时段仅一条排班记录：`(doctor_id, visit_date, time_slot)` 唯一。

### 3.2 冲突检测（按时间段）
- 规则：同一患者同一天同一时段仅允许一条有效预约（避免重复占号）。
- 落点：
  - `appointment` 表建立索引：`(patient_id, visit_date, time_slot)`。
  - 业务层判断：查询是否存在 `status` 属于“仍占用”的记录集合（建议包含 `UNPAID`、`PAID`、`CHECKED_IN`、`COMPLETED`；取消/退款/爽约则不占用）。

### 3.3 支付、退费与状态机
- 预约订单状态：使用字典项 `APPOINTMENT_STATUS`（`UNPAID / PAID / CANCELED / REFUNDED / CHECKED_IN / COMPLETED / NO_SHOW`）。
- 支付与退款分别记录在 `payment_record`、`refund_record` 中，通过预约ID关联，方便对账与审计。

### 3.4 签到入队与排队叫号

| checkin_code              | CHAR(6)  |      | 6位预约签到码  |
| ------------------------- | -------- | ---- | -------------- |
| checkin_code_generated_at | DATETIME |      | 签到码生成时间 |

- 预约订单支付后生成 `checkin_code`。
- 签到成功后插入 `checkin_queue` 记录，生成 `queue_no`，状态变为 `WAITING`（源自字典 `QUEUE_STATUS`）。
- 叫号进度建议使用 `queue_state` 维护“当前叫到几号”，用于患者端快速查询。

### 3.5 临时调班与替诊
- 调班申请记录在 `shift_request`：
  - 影响的排班时段 `target_schedule_id`
  - 接替医生 `replacement_doctor_id`
  - 审批流转 `status`
- 审批通过后：
  - 优先将尚未签到的预约 `appointment.doctor_id` 更新为接替医生；
  - 若无法替换则取消预约并生成退款记录，同时写入提醒日志。

### 3.6 医生评分（仅评分无评价）
- 评分表 `doctor_rating` 约束：
  - 每个预约仅允许评分一次：`appointment_id` 唯一。
- 医生聚合字段：
  - `doctor_profile.avg_rating`、`doctor_profile.rating_count` 用于列表筛选与展示（评分变更后可异步或同步更新）。

### 3.7 数据统计与号源调整推荐（规则型）
- 默认采用实时聚合：对 `appointment`、`doctor_rating` 按日期/科室/医生维度统计。
- 可选扩展每日汇总表 `stat_daily_doctor`，通过定时任务汇总提升查询性能。
- 号源调整推荐：基于历史预约满载情况与爽约率输出建议，不自动改动号源，仅辅助管理员决策。

### 5.5 schedule_slot（排班与号源）
**用途**：按天+时段管理医生出诊与号源容量、剩余量、费用。

| 字段                  | 类型          | 约束     | 说明                                |
| --------------------- | ------------- | -------- | ----------------------------------- |
| id                    | BIGINT        | PK       | 主键                                |
| doctor_id             | BIGINT        | NOT NULL | 医生ID                              |
| dept_id               | BIGINT        | NOT NULL | 科室ID（冗余）                      |
| visit_date            | DATE          | NOT NULL | 日期                                |
| time_slot             | VARCHAR(64)   | NOT NULL | **字典:TIME_SLOT** (AM/PM)          |
| status                | VARCHAR(64)   | NOT NULL | **字典:SCHEDULE_STATUS** (OPEN/...) |
| capacity              | INT           | NOT NULL | 容量                                |
| remaining             | INT           | NOT NULL | 剩余号源                            |
| fee                   | DECIMAL(10,2) | NOT NULL | 挂号费                              |
| open_time/stop_time   | DATETIME      |          | 放号/停号（可选）                   |
| created_by            | BIGINT        |          | 创建人                              |
| created_at/updated_at | DATETIME      |          | 创建/更新时间                       |

**约束与索引**：
- 唯一：`(doctor_id, visit_date, time_slot)`
- 常用：`(dept_id, visit_date)` 用于按科室查号源

**并发控制建议（实现层）**：
- 扣减：`UPDATE ... SET remaining = remaining - 1 WHERE id=? AND remaining>0`
- 回滚：`UPDATE ... SET remaining = remaining + 1 WHERE id=? AND remaining<capacity`

## 6. 索引、约束与典型查询

### 6.1 关键索引总结
1. 冲突检测：`appointment(patient_id, visit_date, time_slot)`
2. 号源唯一：`schedule_slot(doctor_id, visit_date, time_slot)`
3. 队列查询：`checkin_queue(doctor_id, visit_date, time_slot, queue_no)`
4. 医生筛选：`doctor_profile(dept_id)`、`doctor_profile(avg_rating, rating_count)`
5. 统计维度：`appointment(dept_id, visit_date)`、`appointment(doctor_id, visit_date, time_slot)`
6. **字典加速**：`dict_item(type_code, status)` 用于下拉框快速加载。

### 6.2 典型SQL示例
**1）预约冲突检测（同一患者同一天同一时段）**
```sql
SELECT COUNT(1)
FROM appointment
WHERE patient_id = ?
  AND visit_date = ?
  AND time_slot = ?
  AND status IN ('UNPAID','PAID','CHECKED_IN','COMPLETED');
```

**2）扣减号源防超卖（原子更新）**
```sql
UPDATE schedule_slot
SET remaining = remaining - 1,
    updated_at = NOW()
WHERE id = ?
  AND remaining > 0
  AND status = 'OPEN';
```

**3）取消回滚号源**
```sql
UPDATE schedule_slot
SET remaining = remaining + 1,
    updated_at = NOW()
WHERE id = ?
  AND remaining < capacity;
```

**4）生成排队号（示意，实际应事务+锁/序列控制）**
```sql
SELECT IFNULL(MAX(queue_no), 0) + 1
FROM checkin_queue
WHERE doctor_id = ?
  AND visit_date = ?
  AND time_slot = ?;
```

**5）级联查询字典名称（示例）**
```sql
SELECT a.*, di.item_name as status_name
FROM appointment a
LEFT JOIN dict_item di ON a.status = di.item_code
WHERE a.appointment_no = '...';
```

## 7. 权限与数据范围在数据库层的支持

- 管理员数据范围字段位于 `sys_user`：
  - `admin_scope_type`：DEPT/HOSPITAL

  - 科室级：`WHERE dept_id = admin_dept_id`
  - 全院级：不限制或按筛选条件限制

## 8. 备份、恢复与数据安全（毕设写法建议）
1. 定期备份：使用 `mysqldump` 对数据库进行日备份。
2. 敏感字段：手机号、证件号等在展示层脱敏；数据库可不加密但应控制访问权限（论文可描述“可扩展为加密存储”）。
3. 审计：关键操作（调班审批、号源调整、退款）建议在业务日志或操作日志中记录（可作为扩展表）。
